♠️ 打牌记账小程序业务流程与功能拆解
一、 核心业务流程梳理
小程序的整体生命周期可以划分为四个主要阶段：房间准备 -> 游戏记账 -> 结算清盘 -> 数据沉淀。
1. 【创建与准备阶段】
• 房主建房： 用户在首页创建记账房间。
• 邀请入局： 通过小程序卡片分享或二维码邀请好友加入房间。
• 人员就位： 好友点击链接进入房间，房主/用户可长按头像修改昵称（方便现实对号入座）。
• 规则设定： 房主可设置“茶水费”（抽水）规则（如：按赢家比例抽取、设置封顶金额），以及页面布局、屏幕常亮、语音播报等系统辅助功能。
2. 【游戏实时记账阶段】
• 一对一转账： 某局结束后，输家点击赢家头像，输入金额进行定点转账。
• 一对多/多对一（批量记账）： 支持批量操作，例如一炮多响或多人输给庄家时，可一键向多名用户支付或收取相同金额。
• 茶水自动扣除： 若开启了茶水费，每次转账流水产生时，系统根据设定的比例自动计算并划拨部分金额至“茶水”公共账户。
• 流水广播： 所有的记账动作（谁给谁转账、修改了什么规则）都会在房间中央以类似聊天流的形式实时滚动展示，保证公开透明。
• 余额动态更新： 页面顶部各用户的总分（正负收益）实时计算并更新。
3. 【房间结算阶段】
• 发起结算： 游戏结束，房主点击左上角“房间结算”。
• 二次确认： 系统弹窗提示是否解散房间并清算。
• 债务轧差清算（核心环节）： * 系统自动汇总所有人的最终得分（正负）。
• 运行清算算法，得出最优支付方案（如：A 直接给 B 支付 106，茶水费独立结算为 4），将复杂的流水简化为最少次数的最终收付款指令。
4. 【个人数据与历史阶段】
• 数据归档： 房间解散后，对局数据写入用户的个人历史记录。
• 战绩统计： 在“我的”或首页展示用户的历史总场次、胜场次、胜率（如 33.33%）以及累计净收益（如 -140）。
• 流水追溯： 支持查看历史每一局的具体结算时间与盈亏情况，并支持清空历史记录。

二、 关键功能模块拆解
1. 房间管理模块 (Room Management)
• 加入/退出机制： 扫描二维码或点击微信内分享链接加入。
• 自定义昵称： 长按头像修改局内显示名称（不影响微信原生昵称）。
• 茶水（抽水）系统：
• 类型选择：百分比抽水 / 固定满额抽水。
• 参数设置：可设置抽水比例（如 20%）及单次抽水上限（封顶）。
• 房间偏好设置：
• 语音播报开关（增强线下打牌体验，防止错记漏记）。
• 屏幕常亮开关（避免打牌中途手机息屏）。
• 页面布局调整（顶部/左侧/右侧操作区切换）。
2. 记账操作模块 (Accounting Core)
• 快捷小键盘： 点击头像唤起自定义数字小键盘，便于快速输入金额。
• 流水信息流： 按时间倒序排列的全局操作日志区。
• 分发模式：
• 单发（点对点扣减）。
• 群发（批量同步金额给选中人员）。
3. 结算与清算模块 (Settlement & Clearing)
• 实时账单池： 内存中维护所有玩家的当前 Credit (积分)。
• 最终收支清算单： 明确展示“共计支出”、“本局赢家/最终收款人”、“本局输家/最终付款人”及其对应金额，直接指导线下微信/支付宝转账。
4. 个人中心模块 (User Profile)
• 战绩看板： 总场次、胜场次、全局胜率。
• 历史对局列表： 按时间轴展示胜负状态与收支金额。
• 数据清理： 提供清空历史记录的高危操作二次确认

三、 推荐技术架构方案
为了满足“多人同时在线记账、金额实时刷新、操作流公开透明、弱网下仍尽量可用”的目标，推荐采用“微信原生小程序 + Node.js 服务端 + WebSocket 实时推送 + MySQL 持久化 + Redis 房间态缓存”的架构。

1. 总体架构原则
• 服务端作为唯一账本来源： 所有分数、茶水费、结算结果统一由服务端计算并落库，前端不做最终账本裁决，避免多人同时操作造成数据不一致。
• HTTP + WebSocket 双通道： HTTP 负责登录、建房、拉取快照、历史查询；WebSocket 负责房间内实时广播与在线状态同步。
• 事件流 + 快照结合： 每次记账先写流水，再更新房间快照，再推送事件。客户端异常断线后可通过“重新拉取快照 + 继续订阅事件”快速恢复。
• 先做 MVP，再预留扩展： 首版先覆盖单房间实时记账、批量记账、结算清盘；后续再扩展语音播报、分享战绩图、更多牌局规则模板。

2. 推荐技术栈
• 小程序端： 微信原生小程序 + TypeScript + TDesign 小程序组件库。
• 小程序状态管理： `mobx-miniprogram` 或轻量自定义 store，用于维护当前房间快照、成员列表、流水列表、连接状态。
• 服务端： Node.js + NestJS。
• 实时通信： WebSocket。
• 数据库： MySQL 8.x。
• 缓存与房间在线态： Redis。
• 对象存储： 腾讯云 COS，用于头像缓存、结算分享图等静态资源。
• 部署环境： 腾讯云 CVM / 轻量应用服务器，生产建议配合托管版 MySQL 和 Redis。
• 运维组件： Nginx + PM2 / Docker。

推荐理由：
• 微信原生小程序在启动速度、交互响应和微信生态兼容性上更稳，适合你这种“打牌现场高频点击”的产品。
• NestJS 对接口分层、WebSocket 网关、权限拦截、日志审计支持比较完整，后期功能增加时不容易乱。
• MySQL 适合做账目流水、结算记录这类强一致数据。
• Redis 很适合维护房间成员在线状态、热点房间快照和实时广播辅助能力。

3. 系统分层设计
建议拆成以下几层：

• 表现层（小程序端）
  负责页面展示、用户输入、房间实时看板、断线重连、弱网提示、乐观交互反馈。

• 接入层（API Gateway / Nginx）
  负责 HTTPS 接入、鉴权透传、限流、日志记录、反向代理到应用服务。

• 应用层（NestJS）
  建议按业务拆分模块：
  RoomModule：建房、入房、退房、房间配置。
  MemberModule：成员信息、昵称、角色权限、在线状态。
  AccountingModule：单发记账、群发记账、茶水抽成、流水落库。
  SettlementModule：结算、轧差算法、最终支付方案生成。
  HistoryModule：历史局、战绩统计、流水追溯。
  RealtimeGateway：房间内实时广播、连接心跳、重连恢复。
  AuthModule：微信登录、Token 签发、接口鉴权。

• 数据层
  MySQL 负责正式账目与历史数据。
  Redis 负责房间热数据、事件中转、在线人数、幂等键、分布式锁。

4. 实时同步核心设计
这是本项目最关键的一部分，建议按下面方式实现：

• 进入房间时，客户端先通过 HTTP 拉取一次房间完整快照。
  快照内容包括：
  房间信息、成员列表、当前积分、茶水费设置、最新流水、当前版本号。

• 拉取快照成功后，客户端立即建立 WebSocket 连接并订阅当前房间。

• 服务端针对房间内的变化推送标准事件，建议至少包括：
  `member_joined`
  `member_left`
  `score_changed`
  `batch_score_changed`
  `tea_fee_changed`
  `rule_changed`
  `settlement_created`
  `room_dismissed`

• 每个事件都带上：
  `roomId`
  `eventId`
  `roomVersion`
  `serverTime`
  `payload`

• 客户端按 `roomVersion` 顺序应用事件。
  如果发现版本号跳跃，说明有消息丢失，立即重新拉取一次房间快照，保证最终一致。

• WebSocket 断开时自动重连。
  建议采用指数退避，例如 1 秒、2 秒、5 秒、10 秒重试，并在页面上显示“正在重连”状态。

• WebSocket 不可用时降级为轮询。
  例如每 3 到 5 秒拉取一次房间快照，至少保证弱网场景下还能继续看到账目变化。

5. 记账写入链路设计
为了避免多人同时点击导致数据错乱，单次记账应采用统一事务流程：

1. 客户端发起记账请求，携带 `roomId`、操作类型、金额、目标用户、`requestId`。
2. 服务端校验用户身份、房间状态、操作权限。
3. 服务端根据当前规则计算本次记账影响。
4. 开启数据库事务：
   写入账目流水表。
   写入或更新茶水费流水。
   更新房间成员余额快照。
   更新房间版本号。
5. 事务提交成功后，将事件发布到 WebSocket 房间频道。
6. 所有在线客户端收到事件后更新页面。

这里的关键点：
• `requestId` 用于防重复提交，避免用户连点两次造成重复记账。
• `roomVersion` 用于控制事件有序性和客户端补偿同步。
• 账目流水表是最终审计依据，余额快照表只是为了加速展示。

6. 推荐数据模型
建议至少包含以下核心表：

• `users`
  存用户基础信息、微信身份标识、头像昵称。

• `rooms`
  存房间主信息、房主、状态、结算状态、当前版本号。

• `room_members`
  存房间成员、局内昵称、角色、在线状态、当前余额。

• `room_rules`
  存茶水费规则、抽成比例、封顶金额、布局偏好、语音播报等房间配置。

• `ledger_entries`
  存所有记账流水，是最终账本核心表。

• `tea_fee_entries`
  存茶水费或公共费用流水。

• `settlement_records`
  存每次结算结果、轧差方案、结算时间。

• `operation_logs`
  存规则修改、成员加入退出、撤销操作等非账务日志。

• `user_statistics`
  存个人总场次、胜率、累计净收益等汇总数据，可异步更新。

7. 前端页面与数据流建议
为了提升用户体验，小程序端建议采用“快照 + 增量事件”的页面刷新方式，而不是每次整页重刷。

• 首页
  通过 HTTP 获取最近房间、历史战绩摘要。

• 房间页
  首次进入拉全量数据，之后主要依赖 WebSocket 增量更新。
  页面本地维护：
  成员列表
  当前余额
  茶水账户
  流水消息流
  连接状态

• 记账弹层
  点击头像后立即弹出自定义数字键盘。
  用户确认提交后，按钮先进入“提交中”状态，待服务端确认后再刷新最终金额。

• 结算页
  从服务端直接获取清算结果，不在前端本地运算，避免不同客户端算出不同结果。

8. 结算与轧差算法设计
房间结算时建议采用服务端统一清算：

• 先汇总所有成员最终净输赢。
• 将正收益玩家归入收款列表，负收益玩家归入付款列表。
• 通过双指针或贪心法生成最少支付次数的清算方案。
• 茶水费单独列为公共账户结算项，不与玩家个人输赢混算。

这样做的好处是：
• 结算结果稳定统一。
• 可以减少线下实际转账次数。
• 后续容易扩展更多结算规则。

9. 安全与稳定性设计
• 微信登录后，服务端通过 `code2Session` 获取 `openid`，再签发业务 Token。
• 所有写操作接口都必须做身份校验与房间权限校验。
• 重要操作如撤销、结算、解散房间必须写审计日志。
• 接口层增加幂等控制和基础限流，防止重复请求和恶意刷接口。
• Redis 可用于房间级短锁，避免高并发下同一房间多次并发改账。
• 所有客户端错误、WebSocket 断连、服务端异常都需要落日志，方便后期排查。

10. 部署建议
建议分两个阶段推进：

• MVP 阶段
  1 台应用服务器 + 1 个 MySQL 实例 + 1 个 Redis 实例即可满足早期使用。
  适合先验证核心体验：实时记账是否顺滑、房间同步是否稳定、结算方案是否好用。

• 稳定运行阶段
  增加 Nginx 负载均衡，应用服务扩成 2 台以上。
  WebSocket 网关共享 Redis Pub/Sub。
  MySQL 使用托管版并开启自动备份。
  Redis 使用持久化和高可用方案。

11. 为什么这套架构适合当前需求
• 它能满足“实时显示更新”这个核心目标，且不是只刷新页面，而是做到房间级事件推送。
• 它把最终账本放在服务端，能避免多人同时操作时金额不同步的问题。
• 它对后续功能扩展友好，比如增加观战模式、语音播报、战绩榜、分享图都不需要推翻重来。
• 它对微信小程序场景更友好，首屏性能和交互体验会优于偏跨端的重框架方案。

12. 建议的最终落地方案
如果你希望先尽快做出第一版，我建议直接采用下面这个组合：

• 前端： 微信原生小程序 + TypeScript + TDesign。
• 后端： NestJS + WebSocket + MySQL + Redis。
• 部署： 腾讯云服务器 + Nginx + PM2。

这是一套兼顾开发效率、实时能力、后续可扩展性和维护成本的方案，比较适合你这个“多人在线实时记账”的微信小程序项目。
