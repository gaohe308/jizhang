# 打牌记账小程序后端开发计划

## 1. 当前确定的技术路线

为了尽快上线一个稳定、可维护、同时开发成本可控的 MVP 版本，当前后端方案确定为：

- 前端：微信原生小程序 + TypeScript + TDesign
- 后端：NestJS
- 数据库：MySQL
- 部署：CloudBase 云托管
- 实时更新：单实例 WebSocket
- Redis：第一版先不引入，后续视需要再补

这套方案的核心思路是：

- 先保证核心业务稳定可用
- 先把正式账本放在服务端统一管理
- 先做能上线的最小版本
- 避免一开始就引入过多基础设施

## 2. 为什么 MVP 先不上 Redis

当前项目虽然有“多人实时记账”的需求，但首版 MVP 的使用规模和并发量都不会太高。

在这个阶段：

- 单实例 NestJS + MySQL + WebSocket 已经足够支撑房间内的实时更新
- Redis 并不是首版上线的硬性依赖
- 不引入 Redis 可以减少本地开发、联调、部署和维护复杂度

Redis 在后续更适合用于这些场景：

- 多实例 WebSocket 广播
- 房间在线态共享
- 幂等键集中管理
- 热点房间缓存
- 服务横向扩容

所以当前结论是：

- MVP 第一版：不引入 Redis
- 第二阶段：如果房间实时更新和部署规模上来，再引入 Redis

## 3. 后端总体架构

### 3.1 开发阶段

当前本地开发阶段使用：

- 本地 NestJS
- 本地 MySQL
- 本地 WebSocket
- Redis 暂不接入

### 3.2 上线阶段

正式上线后使用：

- CloudBase 云托管部署 NestJS
- CloudBase MySQL 作为正式数据库
- 单实例 WebSocket 先支撑房间实时更新
- Redis 暂不接入，等需要时再接入 CloudBase Redis

## 4. 当前前端功能对应的后端能力

结合目前已完成的前端页面，后端需要支撑以下功能。

### 4.1 首页

- 创建房间
- 扫码 / 房间号加入房间
- 获取本周战绩摘要

### 4.2 房间页

- 拉取房间快照
- 获取成员列表、余额、茶水账户、规则、流水
- 单人记账
- 批量记账
- 修改局内昵称
- 修改茶水规则
- 修改房间偏好设置
- 实时更新房间余额和流水

### 4.3 结算页

- 获取服务端生成的结算结果
- 获取茶水账户结算项
- 归档并解散房间

### 4.4 个人中心

- 获取总场次、胜场、胜率、累计净收益
- 获取历史记录
- 清空历史记录

## 5. 核心业务规则

### 5.1 茶水费规则

当前后端逻辑与现有前端保持一致：

- 每次记账时按房间规则计算茶水
- 茶水费从收款方到账金额中扣除
- 茶水账户独立累计
- 结算时茶水账户单独展示，不并入玩家互相轧差

也就是：

- 付款方支付原始金额
- 收款方拿到扣除茶水后的净额
- 茶水账户获得茶水金额

### 5.2 茶水模式

支持两种模式：

- 百分比抽水
- 满额抽水

共同规则：

- 每笔茶水都受封顶值限制

### 5.3 结算规则

- 玩家之间做债务轧差
- 茶水账户单独展示
- 最终结算方案由服务端统一生成

### 5.4 权限规则

- 房主可修改房间规则
- 房主可发起归档结算
- 成员可修改自己的局内昵称
- 非房间成员不能访问房间数据

## 6. 后端模块划分

### 6.1 AuthModule

职责：

- 微信登录
- `code2Session`
- JWT 签发
- 登录态校验

接口：

- `POST /api/auth/login`

### 6.2 UserModule

职责：

- 获取当前用户
- 获取个人统计
- 获取历史记录
- 清空历史记录

接口：

- `GET /api/users/me`
- `GET /api/users/me/stats`
- `GET /api/users/me/history`
- `DELETE /api/users/me/history`

### 6.3 RoomModule

职责：

- 创建房间
- 加入房间
- 获取房间快照

接口：

- `POST /api/rooms`
- `POST /api/rooms/join`
- `GET /api/rooms/:roomId`

### 6.4 MemberModule

职责：

- 修改局内昵称

接口：

- `PATCH /api/rooms/:roomId/members/me/nickname`

### 6.5 RuleModule

职责：

- 修改茶水规则
- 修改布局
- 修改房间偏好

接口：

- `PATCH /api/rooms/:roomId/rules`

### 6.6 AccountingModule

职责：

- 单人记账
- 批量记账
- 茶水计算
- 流水落库
- 更新房间余额

接口：

- `POST /api/rooms/:roomId/accounting/single`
- `POST /api/rooms/:roomId/accounting/batch`

### 6.7 SettlementModule

职责：

- 结算预览
- 最少支付次数方案生成
- 归档房间
- 写历史记录

接口：

- `GET /api/rooms/:roomId/settlement`
- `POST /api/rooms/:roomId/settlement/archive`

### 6.8 RealtimeGateway

职责：

- WebSocket 房间订阅
- 广播房间余额变化
- 广播流水变化
- 广播规则变化

首版事件建议：

- `room.snapshot`
- `member.joined`
- `member.updated`
- `rule.updated`
- `ledger.created`
- `balance.updated`
- `settlement.created`
- `room.archived`

## 7. 数据库表结构

首版使用 MySQL 保存正式业务数据。

### 7.1 核心表

- `users`
- `rooms`
- `room_members`
- `room_rules`
- `ledger_entries`
- `ledger_entry_items`
- `settlements`
- `settlement_items`
- `history_records`
- `idempotency_records`

### 7.2 表结构设计原则

- MySQL 中的账务流水是最终审计依据
- 房间余额是快照数据，用于加速展示
- 历史记录是归档结果，不参与当前房间运算
- `request_id` 用于控制重复提交

## 8. 实时更新方案

MVP 第一版的实时更新方案如下：

1. 前端进入房间时先拉一次 HTTP 快照
2. 再建立 WebSocket 连接
3. 订阅当前房间
4. 服务端在记账、改规则、改昵称、结算时主动广播事件
5. 前端收到事件后刷新成员余额、流水列表和页面状态

### 8.1 为什么首版不用 Redis 也能做

因为当前首版默认只有一个后端实例，WebSocket 连接和业务写入都在同一个 NestJS 进程里。

在这种情况下：

- 写入成功后直接在当前进程广播房间事件即可
- 不需要 Redis Pub/Sub

### 8.2 后续什么时候引入 Redis

当出现这些情况时再引入 Redis：

- 云托管服务扩成多实例
- 需要跨实例同步房间事件
- 房间在线态要统一管理
- 幂等控制和缓存需求明显增加

## 9. 写入事务链路

所有写操作按统一流程执行：

1. 校验登录态
2. 校验房间状态
3. 校验操作身份
4. 校验参数
5. 校验 `requestId`
6. 开启数据库事务
7. 写入流水和明细
8. 更新成员余额
9. 更新房间版本号
10. 提交事务
11. 广播 WebSocket 事件

核心原则：

- 服务端是唯一账本来源
- 所有写操作必须幂等
- MySQL 保存最终真相

- 建立数据库模型
- 建立环境变量配置

### 第三阶段：基础业务模块

- 登录
- 用户
- 房间
- 成员
- 规则

### 第四阶段：账务核心

- 单人记账
- 批量记账
- 茶水计算
- 流水落库

### 第五阶段：实时更新

- 接入单实例 WebSocket

### 第六阶段：结算与历史

- 结算预览

1. 首页和个人页只读接口
2. 房间快照接口
3. 规则接口
4. 单人记账接口
5. 批量记账接口
6. 结算和归档接口
7. WebSocket 实时更新

### 第八阶段：上线

- 部署到 CloudBase 云托管
- 正式数据库切 CloudBase MySQL
- 首版不接 Redis
- 小程序切线上接口
- 提交微信审核

## 11. MVP 验收标准

满足以下条件即可认为 MVP 达标：

- 用户可以登录并创建房间
- 多个用户可以加入同一房间
- 房间内可以完成单人记账
- 批量记账可以正确落账
- 茶水规则正确生效
- 房间流水和余额可以实时更新
- 结算方案由服务端统一生成
- 房间归档后历史和统计正确更新

当前后端开发按下面顺序继续：

1. 完成 `backend` 依赖安装
2. 执行 Prisma 初始化与迁移
3. 跑通本地 MySQL
4. 接着开发 WebSocket 实时更新
5. 再进入前端联调
