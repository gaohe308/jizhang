# 前后端联调测试清单

## 1. 测试目标

本清单用于验证当前项目的本地开发环境是否已经具备以下能力：

- 后端可以正常启动
- 本地 MySQL 已完成 Prisma 迁移
- 核心接口可以正常访问
- 房间、成员、规则、记账、结算主链路可跑通
- 为后续前端替换 `mock.ts` 做联调准备

## 2. 当前代码现状说明

当前后端已经具备以下模块和接口骨架：

- 健康检查
- 微信登录
- 当前用户信息 / 统计 / 历史
- 创建房间 / 加入房间 / 获取房间快照
- 修改局内昵称
- 修改房间规则
- 单人记账
- 批量记账
- 结算预览
- 归档房间

当前还未完成或未接入的部分：

- 前端真实接口替换
- WebSocket 实时广播
- Redis 实际业务接入

所以这份测试清单分为两部分：

- 后端接口自测
- 前端联调前准备检查

## 3. 测试前准备

### 3.1 基础环境

确认以下环境已就绪：

- Node.js 可用
- 本地 MySQL 可用
- 本地数据库 `poker_bookkeeping` 已创建
- `backend/.env` 已配置
- Prisma 已执行过迁移
- 后端已能本地启动

### 3.2 启动命令

启动本地 Redis：

```powershell
powershell -ExecutionPolicy Bypass -File e:\w\wx_project\jizhang\scripts\start-redis.ps1
```

启动本地后端：

```powershell
powershell -ExecutionPolicy Bypass -File e:\w\wx_project\jizhang\scripts\start-backend.ps1
```

停止本地后端：

```powershell
powershell -ExecutionPolicy Bypass -File e:\w\wx_project\jizhang\scripts\stop-backend.ps1
```

### 3.3 健康检查

执行：

```powershell
Invoke-WebRequest http://127.0.0.1:3000/api/health -UseBasicParsing
```

预期结果：

- 返回 `200`
- body 中包含：
  - `ok: true`
  - `service: poker-bookkeeping-backend`

## 5. 登录链路测试

### 5.1 当前说明

当前登录接口依赖微信 `code2Session`，所以要测试成功，必须拿到一个真实可用的小程序登录 `code`。

接口位置：

- `POST /api/auth/login`

### 5.2 测试方法

前端小程序中调用 `wx.login()`，拿到 `code` 后，使用该 `code` 请求：

```http
POST http://127.0.0.1:3000/api/auth/login
Content-Type: application/json

{
  "code": "真实的微信登录code"
}
```

预期结果：

- 返回 `token`
- 返回 `user.id`
- `users` 表新增或更新一条用户记录

### 5.3 注意事项

如果返回微信登录失败，不一定是代码问题，优先检查：

- `WECHAT_APP_ID`
- `WECHAT_APP_SECRET`
- 当前 `code` 是否过期

## 6. 鉴权接口通用测试方式

登录成功后，把返回的 `token` 记下来，后续所有需要登录的接口都要加：

```http
Authorization: Bearer 你的token
```

## 15. 当前版本的已知限制

当前版本测试时请注意这些限制：

- 登录必须依赖真实微信 `code`
- 还没有接前端真实接口
- 还没有接 WebSocket 实时推送
- 还没有引入 Redis
- 还没有做接口自动化测试

也就是说：

- 当前重点先测“后端接口和数据库链路是否正确”
- 下一步再测“前端页面与后端接口是否能完全联动”

## 16. 建议测试顺序

建议你按这个顺序测试：

1. 健康检查
2. 数据库检查
3. 登录
4. 创建房间
5. 获取房间快照
6. 修改昵称
7. 修改规则
8. 单人记账
9. 批量记账
10. 结算预览
11. 房间归档
12. 用户统计和历史

如果这 12 步都通过，就说明当前前后端主业务链已经基本跑通。
